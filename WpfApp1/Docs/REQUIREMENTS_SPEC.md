# 요구사항 명세서

문서명: 스마트 자동화 설비 통합 제어 소프트웨어 요구사항 명세서  
문서 버전: v0.1  
최종 수정일: 2026-04-14  
기준 문서: `PROJECT_PLAN.md`

## 1. 문서 목적

본 문서는 프로젝트의 기능적 요구사항과 비기능 요구사항을 정의한다.  
현재 기준은 `WPF 중심 1차 개발 범위`이며, 이후 하드웨어 연동 요구사항은 확장 항목으로 관리한다.

## 2. 시스템 범위

본 시스템은 제조 자동화 환경에서 사용하는 PC 기반 운영 소프트웨어를 대상으로 한다.

현재 범위:

- 대시보드
- 검사 이력 관리
- 알람 로그 관리
- 레시피 관리
- 수동 제어 화면
- Mock 데이터 기반 상태 시뮬레이션

향후 확장 범위:

- Jetson Orin Nano 연동
- OAK-D Lite 연동
- Arduino / 페어리노 연동
- Dobot 연동

## 3. 사용자 정의

### 3.1 운영자

- 시스템 상태를 확인한다.
- 생산 및 검사 결과를 조회한다.
- 수동 제어를 수행한다.
- 알람을 확인한다.

### 3.2 관리자

- 운영자 기능을 모두 수행할 수 있다.
- 레시피를 등록/수정/삭제한다.
- 시스템 설정을 변경한다.
- 통신 및 장비 설정을 관리한다.

## 4. 기능 요구사항

## FR-01 대시보드 조회

설명: 사용자는 메인 화면에서 전체 시스템의 현재 상태를 즉시 확인할 수 있어야 한다.

세부 요구:

- 설비 상태를 표시해야 한다.
- 자동 / 수동 / 정지 상태를 표시해야 한다.
- 생산 수량, 정상 수량, 불량 수량을 표시해야 한다.
- 장비 연결 상태를 표시해야 한다.
- 최근 알람 요약을 표시해야 한다.

우선순위: 높음

## FR-02 검사 결과 표시

설명: 사용자는 현재 검사 결과와 최근 검사 정보를 확인할 수 있어야 한다.

세부 요구:

- 현재 검사 결과를 `OK / NG` 형태로 표시해야 한다.
- 마지막 검사 시각을 표시해야 한다.
- 제품 ID 또는 검사 대상 식별값을 표시해야 한다.
- 결과에 따라 화면 강조 표현이 달라져야 한다.

우선순위: 높음

## FR-03 검사 이력 조회

설명: 사용자는 이전 검사 결과를 목록으로 조회할 수 있어야 한다.

세부 요구:

- 검사 시각, 제품 ID, 결과, 비고를 저장해야 한다.
- 기간 조건으로 조회할 수 있어야 한다.
- 결과 조건(`OK`, `NG`, 전체)으로 필터링할 수 있어야 한다.
- 최신 데이터가 상단에 표시되어야 한다.

우선순위: 높음

## FR-04 알람 로그 조회

설명: 사용자는 시스템에서 발생한 알람과 이벤트를 확인할 수 있어야 한다.

세부 요구:

- 알람 구분(정보, 경고, 오류)을 표시해야 한다.
- 발생 시각과 메시지를 표시해야 한다.
- 확인 여부를 관리할 수 있어야 한다.
- 최신 알람이 상단에 표시되어야 한다.

우선순위: 높음

## FR-05 레시피 관리

설명: 관리자는 제품별 설정값을 레시피로 관리할 수 있어야 한다.

세부 요구:

- 레시피 목록을 조회할 수 있어야 한다.
- 신규 레시피를 등록할 수 있어야 한다.
- 기존 레시피를 수정할 수 있어야 한다.
- 레시피를 삭제할 수 있어야 한다.
- 레시피에는 제품명, 검사 기준값, 공정 설정값이 포함되어야 한다.

우선순위: 높음

## FR-06 수동 제어

설명: 사용자는 화면에서 기본 제어 명령을 수행할 수 있어야 한다.

세부 요구:

- 시작, 정지, 리셋 버튼을 제공해야 한다.
- 자동 / 수동 모드를 전환할 수 있어야 한다.
- 개별 장치 제어를 위한 버튼 영역을 제공해야 한다.
- 현재는 Mock 동작으로 처리하고, 이후 실제 장비 명령으로 대체 가능해야 한다.

우선순위: 높음

## FR-07 상태 시뮬레이션

설명: 하드웨어 연결 전에도 시스템 검증이 가능해야 한다.

세부 요구:

- Mock 검사 결과를 생성할 수 있어야 한다.
- Mock 장비 연결 상태를 변경할 수 있어야 한다.
- Mock 알람을 발생시킬 수 있어야 한다.
- Mock 데이터는 실제 연동 서비스와 교체 가능한 구조여야 한다.

우선순위: 높음

## FR-08 통신 상태 표시

설명: 사용자는 장비별 통신 상태를 확인할 수 있어야 한다.

세부 요구:

- 장비명과 연결 상태를 표시해야 한다.
- 마지막 수신 시각을 표시할 수 있어야 한다.
- 현재 단계에서는 Mock 상태를 표시한다.

우선순위: 중간

## FR-09 데이터 저장

설명: 시스템은 주요 운영 데이터를 로컬 저장소에 보관해야 한다.

세부 요구:

- 검사 이력을 저장해야 한다.
- 알람 로그를 저장해야 한다.
- 레시피 정보를 저장해야 한다.
- 앱 재시작 후에도 저장 데이터가 유지되어야 한다.

우선순위: 높음

## FR-10 통계 표시

설명: 사용자는 생산 및 검사 집계 정보를 확인할 수 있어야 한다.

세부 요구:

- 총 생산 수량을 표시해야 한다.
- OK 수량과 NG 수량을 표시해야 한다.
- 기본 차트 또는 요약 카드 형태로 표현할 수 있어야 한다.

우선순위: 중간

## 5. 비기능 요구사항

## NFR-01 사용성

- 산업용 운영 화면을 전제로 한 명확한 정보 배치가 필요하다.
- 주요 상태 정보는 한눈에 식별 가능해야 한다.
- 경고/오류 상태는 색상 또는 강조 요소로 구분 가능해야 한다.

## NFR-02 확장성

- 장비 연동은 인터페이스 기반 구조여야 한다.
- Mock 구현체와 실제 구현체를 교체 가능해야 한다.
- 화면 구조는 기능 추가 시 확장이 가능해야 한다.

## NFR-03 유지보수성

- MVVM 패턴을 적용해야 한다.
- UI, 서비스, 저장소, 도메인 모델을 분리해야 한다.
- 공통 상태와 이벤트 처리 로직은 재사용 가능해야 한다.

## NFR-04 성능

- 기본 화면 전환은 즉시 반응해야 한다.
- 대시보드 상태 갱신은 사용자가 지연을 체감하지 않는 수준이어야 한다.
- 목록 조회는 일반적인 로컬 데이터 규모에서 원활해야 한다.

## NFR-05 신뢰성

- 예외 발생 시 앱이 즉시 종료되지 않도록 처리해야 한다.
- 주요 이벤트는 로그로 남겨야 한다.
- 잘못된 입력에 대한 검증이 필요하다.

## NFR-06 저장성

- 로컬 DB에 저장된 데이터는 앱 종료 후에도 유지되어야 한다.
- 레시피 및 로그는 재실행 후 재조회 가능해야 한다.

## 6. 데이터 요구사항

### 6.1 검사 이력

- ID
- 검사 시각
- 제품 ID
- 결과
- 판정값 또는 비고

### 6.2 알람 로그

- ID
- 발생 시각
- 레벨
- 메시지
- 확인 여부

### 6.3 레시피

- ID
- 레시피명
- 제품명
- 기준값
- 공정 옵션
- 사용 여부

### 6.4 장비 상태

- 장비명
- 연결 상태
- 현재 상태
- 마지막 갱신 시각

## 7. 제약사항

- 초기 개발은 실제 하드웨어 없이 진행한다.
- 1차 단계의 핵심은 WPF 애플리케이션 구조와 사용자 경험이다.
- 장비 통신 방식은 실제 연동 시점에 확정한다.
- 현재는 단일 PC 환경 기준으로 설계한다.

## 8. 향후 확장 요구사항

### 8.1 비전 연동

- Jetson으로부터 검사 결과를 수신해야 한다.
- OAK-D Lite의 이미지 또는 판정 정보를 표시할 수 있어야 한다.

### 8.2 I/O 연동

- Arduino / 페어리노와 Serial 또는 TCP 통신이 가능해야 한다.
- 센서 입력 상태를 표시해야 한다.
- 출력 제어 명령을 송신할 수 있어야 한다.

### 8.3 로봇 연동

- Dobot 상태를 조회할 수 있어야 한다.
- 명령 송신 인터페이스를 둘 수 있어야 한다.
- 검사 결과에 따라 이송 / 분류 시퀀스를 트리거할 수 있어야 한다.

## 9. 수용 기준 초안

- 대시보드에서 상태 카드와 생산 집계가 정상 표시된다.
- 검사 이력 등록 후 목록에서 조회된다.
- 알람 발생 시 로그 목록에 저장되고 표시된다.
- 레시피를 등록/수정/삭제할 수 있다.
- 수동 제어 버튼 입력 시 Mock 상태가 변경된다.
- 앱 종료 후 재실행해도 저장 데이터가 유지된다.

## 10. 변경 관리

- 요구사항 추가 시 식별자(`FR`, `NFR`)를 부여한다.
- 우선순위 변경 시 본 문서를 기준으로 관리한다.
- 구현 전 요구사항 변경이 발생하면 `PROJECT_PLAN.md`와 함께 동기화한다.
